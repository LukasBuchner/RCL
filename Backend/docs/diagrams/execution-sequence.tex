\subsection{Sequence Diagram Analysis}

The sequence diagram in Figure~\ref{fig:execution-sequence} illustrates the temporal ordering of messages in the execution pipeline.

\subsubsection{Initialization Phase (Steps 1--2)}
Before execution begins, the reactive subscriptions are established:
\begin{enumerate}
    \item \textbf{Trigger Service} subscribes to the Event Bus's \texttt{AllEvents} observable
    \item \textbf{Execution Orchestrator} subscribes to the same observable
\end{enumerate}
These subscriptions remain active throughout the execution lifetime.

\subsubsection{Inner Loop Execution (Steps 3--9)}
The inner loop executes without throttling for fast event propagation:
\begin{enumerate}
    \setcounter{enumi}{2}
    \item Trigger Service detects that prerequisites are met and calls \texttt{TriggerSkill()}
    \item Coordinator calls \texttt{ExecuteSkillAsync()} on the Runtime Agent
    \item Agent returns an \texttt{IObservable<Progress>} stream
    \item On first progress, Coordinator pushes \texttt{Start} event via \texttt{OnNext}
    \item Agent completes execution with \texttt{OnCompleted}
    \item Coordinator pushes \texttt{Finish} event via \texttt{OnNext}
    \item Event Bus emits to Trigger Service, potentially triggering next skill
\end{enumerate}

\subsubsection{Outer Loop Processing (Steps 10--16)}
The outer loop handles state management and rescheduling with throttling:
\begin{enumerate}
    \setcounter{enumi}{9}
    \item Event Bus emits to Orchestrator (same event, different subscriber)
    \item Orchestrator calls State Transition Service to update execution state
    \item After 500ms throttle window, Orchestrator calls \texttt{RescheduleAsync()}
    \item Rescheduler queries State Manager for current execution states
    \item Rescheduler returns updated schedule with new timing
    \item Orchestrator broadcasts updated nodes to Frontend via GraphQL
    \item For adaptive skills, Orchestrator updates planned finish times
\end{enumerate}

\subsubsection{Key Observations}

\paragraph{Parallel Subscription}
Both the Trigger Service and Orchestrator subscribe to the same Event Bus observable.
When an event is emitted, \emph{both} receive it simultaneously, enabling:
\begin{itemize}
    \item Trigger Service to immediately check for next triggerable skills (fast)
    \item Orchestrator to update state and schedule rescheduling (throttled)
\end{itemize}

\paragraph{Stream vs.\ Call Semantics}
The diagram distinguishes between:
\begin{itemize}
    \item \textbf{Synchronous calls} (solid arrows): Traditional method invocations that block until completion
    \item \textbf{Stream emissions} (orange dashed): Rx.NET \texttt{OnNext} calls that push data into observables
    \item \textbf{Subscriptions} (blue dashed): Rx.NET \texttt{Subscribe} calls that establish reactive connections
\end{itemize}

\paragraph{Throttling Boundary}
The 500ms \texttt{Sample} operator creates a clear boundary:
\begin{itemize}
    \item Events arriving within the window are coalesced
    \item Only the most recent event triggers rescheduling
    \item This prevents scheduler overload during rapid event sequences
\end{itemize}
